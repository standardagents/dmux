Architecture Approaches

How to wire task adherence analysis + dashboard mode into the existing codebase

Subsystem 1: Analysis Pipeline (Task Adherence)

Adding "on track" evaluation to the existing PaneAnalyzer → StatusDetector flow

A

Extend PaneAnalyzer Stage 3

Add task adherence fields to the existing Stage 3 (summary extraction) LLM call. The prompt already receives pane content — we add the stored task context and ask for onTrack + confidence alongside the existing summary.

Pros

  • No extra LLM calls — same latency & cost
  • Summary and adherence evaluated together = coherent output
  • Minimal new code — extends existing interface

Cons

  • Stage 3 prompt gets more complex
  • If task context is long, may exceed token budget (currently 180 tokens output)
  • Adherence eval only runs when pane settles (idle/prompt), not while working
B

New Stage 4: Periodic Adherence Check

Add a separate analysis stage that runs on a timer (e.g., every 30-60s) for panes in working state. Uses stored task context + recent output to evaluate adherence independently of the idle/prompt detection pipeline.

Pros

  • Catches drift while agent is still working, not just when idle
  • Independent cadence — doesn't bloat existing analysis stages
  • Can use a cheaper/faster model for this specific task

Cons

  • Extra LLM calls = higher cost
  • New timer + cancellation logic in StatusDetector
  • Two separate summaries to reconcile (adherence vs idle summary)
C

Hybrid: Extend Stage 3 + Periodic Working Check

Extend Stage 3 for idle/prompt states (approach A). Additionally, run a lightweight periodic check for working panes that only returns adherence (no full summary). Two paths optimized for their context.

Pros

  • Catches drift in both working and idle states
  • Working check is lightweight (just on-track + confidence)
  • Idle summary remains coherent with adherence

Cons

  • Most complex implementation
  • Two code paths to maintain
  • Still has LLM cost for periodic checks

Subsystem 2: Dashboard Mode (Layout/Visibility)

How the user-toggled dashboard mode interacts with existing hide/show and layout systems

A

Reuse Existing Hidden Flag

Dashboard mode = bulk-hide all working panes using the existing hidden flag and paneVisibility utilities. Auto-expand = unhide. The layout system already handles hidden panes (they're excluded from grid calculation).

Pros

  • Minimal layout changes — hidden panes already excluded from grid
  • Existing hide/show actions (h key) still work
  • PaneCard already shows "(hidden)" indicator

Cons

  • No distinction between "user hid this" and "dashboard auto-hid this"
  • Toggling dashboard mode off would need to restore only auto-hidden panes
  • Need to track which panes were auto-hidden vs manually hidden
B

New "minimized" State (Separate from Hidden)

Add a minimized boolean to DmuxPane, distinct from hidden. Dashboard mode sets minimized: true on working panes. The layout system treats minimized panes like hidden (excluded from grid), but the sidebar renders them differently — with the enhanced summary line instead of "(hidden)".

Pros

  • Clean separation: hidden = user choice, minimized = dashboard automation
  • Toggling dashboard off restores only minimized panes
  • PaneCard can render minimized state distinctly (summary line)
  • Manual hide (h key) still works independently

Cons

  • New field on DmuxPane type
  • Layout calculator needs to exclude both hidden and minimized
  • Config file grows slightly

Recommended Combination

Analysis C (Hybrid) + Layout B (minimized state)

The hybrid approach gives you drift detection while agents work (the whole point of dashboard mode) plus coherent summaries when they stop. The separate minimized state keeps dashboard automation cleanly separated from manual hide/show, making the toggle predictable.
